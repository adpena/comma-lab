## Critical hardware-reality finding

**NVFP4 hardware requires Blackwell (compute capability 10.0). RTX 4090 is CC 8.9.** Every Lane F result (V1=2.73, V2=1.79, V3=1.85, V4 in flight) was generated by FakeQuantFP4 simulation — software emulation with NO hardware FP4 backing. The 20× PoseNet penalty we attributed to "FP4 architectural hostility" may be entirely an artifact of the simulation's numerics, not the real quantization scheme.

**FP8 IS hardware-supported on 4090** via `torchao.float8.convert_to_float8_training`. Cosmos RL's `ROWWISE_WITH_GW_HP` recipe keeps grad_weight at high precision while quantizing forward + grad_input — directly addresses the PoseNet penalty mechanism by NOT quantizing the tiny accumulated gradient signals that FP4 destroyed.

**This invalidates several earlier conclusions:**
- "FP4 is architecturally hostile" → unverified; was simulated only
- "Don't FP4 other archs without sensitivity test" → should be "Don't simulated-FP4; do hardware-FP8 instead"
- Lane F-V4 mixed-precision FP4 (in flight on Vast.ai) → should be retargeted to FP8 if it lands negative

## 8 transferable patterns extracted from actual runnable code

### Pattern 1 — Multi-control hint encoder (`general_dit_ctrl_enc.py`)
- Per-modality MLP `hint_nf=[16,16,32,32,96,96,256]` → `zero_module` final projection → per-block summation with per-pixel `control_weight` map
- ~6K extra params for 3 modalities — fits our scale
- Direct application: replace our scalar `mask_concat` conditioning with a per-pixel control-weight modulation

### Pattern 2 — Multi-modality fusion (`world_generation_pipeline.py`)
- Modalities are normalized to a common feature space via per-modality `norm_layer` then summed (not concatenated)
- Lower rate cost than concatenation-based fusion at our scale

### Pattern 3 — Cosmos RL FP4/FP8 protected-ops recipe (`cosmos-rl/utils/fp4/`, ~900 LOC)
- `ROWWISE_WITH_GW_HP`: forward + grad_input quantized to FP4/FP8; grad_weight kept at FP16/BF16
- Per-tensor scaling NOT per-channel (counter-intuitive given Lane S uses per-channel)
- Calibration via running mean/abs-max with momentum 0.95
- Activation scaling computed at training start, FROZEN thereafter
- This is the recipe to apply for Lane F-V5

### Pattern 4 — HighSigmaStrategy (Cosmos Predict 2.5 action conditioner)
- During training, 5% of samples have sigma redrawn from [80, 2000] (vs normal [0.5, 80] range)
- Forces model to learn out-of-distribution noise levels
- Closes proxy/auth gap from a SECOND axis orthogonal to Lyra's input-perturbation SAUG

### Pattern 5 — Soft-DTW outlier scoring (`cosmos-cookbook/data_curation/outlier_detection.md`, 479 LOC)
- SegNet encoder → PCA-3 dimensionality reduction → Time Series K-Means clustering → soft-DTW barycenter distance per pair → Q95 per-cluster outlier threshold
- INDEPENDENT typicality signal — replaces Lane W's circular "weight by own loss" approach
- Compress-time only — Curator architecture fits our budget

### Pattern 6 — 3D Haar wavelet (Cosmos Tokenizer)
- Parameter-free, invertible, analytical
- Free multi-scale front-end on the masks input
- Could replace some learned conv layers with no rate cost

### Pattern 7 — CFG-style triple-dropout
- Training: text=0.2 dropout, video=0.2, ctrl=0.5 simultaneously
- Makes one model handle all CFG combinations
- For us: train renderer with random "no mask" / "no pose" / "no embedding" forward passes

### Pattern 8 — Inpainting hook in `denoise()`
- `raw_x0 = guided_mask × guided_image + (1-guided_mask) × x0`
- Architectural primitive for "lock these pixels, regenerate those"
- Could selectively regenerate only PoseNet-critical regions
