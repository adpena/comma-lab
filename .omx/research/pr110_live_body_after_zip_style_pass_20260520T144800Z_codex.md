# changes from upstream

FEC6 selector bolt-on around the public HNeRV lineage. The HNeRV decoder in `src/model.py` originates in PR [#95](https://github.com/commaai/comma_video_compression_challenge/pull/95) by `@AaronLeslie138` (`hnerv_muon`) and is byte-identical here. The immediate byte substrate is PR [#101](https://github.com/commaai/comma_video_compression_challenge/pull/101) by `@SajayR` (`hnerv_ft_microcodec`), inheriting its compact decoder, latent-payload parsing, Brotli-coded source streams, and canonical Huffman for the latent sidecar. Prior lineage: PR [#98](https://github.com/commaai/comma_video_compression_challenge/pull/98) (`@EthanYangTW`), PR [#100](https://github.com/commaai/comma_video_compression_challenge/pull/100) (`@BradyMeighan`), PR [#102](https://github.com/commaai/comma_video_compression_challenge/pull/102) (`@EthanYangTW`), PR [#103](https://github.com/commaai/comma_video_compression_challenge/pull/103) (`@rem2`). This submission does not import PR #103's `constriction` range coder.

New in this submission: a 31-mode per-frame transform selector (K=16 active palette) chosen offline against the upstream scorer + a fixed-Huffman k=16 codebook over the selector indices. Selector framework is ~265 LOC of Python (`src/frame_selector.py`). Member `x` wraps the PR #101 source payload as `FP11 + source_len + source_pr101_payload + selector_len + selector_payload`; PR #101's Brotli envelope is unchanged and operates only inside the source-payload region, not at the ZIP layer and not over the appended FEC6 selector bitstream. Archive byte delta vs PR #101 source is +259 bytes (rate cost `25 * 259 / 37545489 = 0.000172`).

Submitted by Alejandro Peña <adpena@gmail.com>. Submission name follows the `hnerv_<descriptor>` convention from PR #95 `hnerv_muon` / PR #101 `hnerv_ft_microcodec`. Thanks to @YassineYousfi for clarifying the late-submission rubric in the PR [#108](https://github.com/commaai/comma_video_compression_challenge/pull/108) discussion. **Where this sits.** This submission improves the immediate HNeRV-family byte substrate I used for this submission: PR #101 `0.1928450127024255 [contest-CPU]` -> PR #110 `0.1920513168811056 [contest-CPU]` (`-0.0007936958213199`, reported as `-0.000794`, including the +259-byte rate cost). The broader HNeRV lineage is related but should not be collapsed into that single delta; PR #95 / #98 / #100 / #102 / #103 use related decoder and sidecar ideas with different payloads and reported scores. Paired CUDA is `0.22621002169349796 [contest-CUDA T4]`, reported separately from the CPU axis.

In my local experiments on this archive and related HNeRV variants, running general-purpose compressors on the exported `archive.zip` member bytes did not produce a material score reduction. Further score reductions will likely require changing the emitted representation, training priors/objectives, entropy model, archive grammar, or decoder-side side information rather than post-hoc recompressing the same bytes.

For broader research context, I am keeping substrate-shift details outside this submission PR. [`adpena/comma-lab`](https://github.com/adpena/comma-lab) contains ongoing research notes, candidate inventories, and source maps, and [`adpena/tac`](https://github.com/adpena/tac) contains reusable research tooling. Those repositories are references for follow-up work only; the submitted runtime in this PR is self-contained aside from stdlib, `torch`, `numpy`, and `brotli`, and does not depend on those repositories.
