# changes from upstream

This submission adds **two new bolt-ons** on top of the public HNeRV substrate established by @SajayR (PR [#95](https://github.com/commaai/comma_video_compression_challenge/pull/95) decoder + PR [#101](https://github.com/commaai/comma_video_compression_challenge/pull/101) compact fine-tuned HNeRV microcodec, current top merged at `0.192845 [contest-CPU]`) and the subsequent medal-class iterations by @AaronLeslie138 ([#98](https://github.com/commaai/comma_video_compression_challenge/pull/98)), @EthanYangTW ([#100](https://github.com/commaai/comma_video_compression_challenge/pull/100), [#102](https://github.com/commaai/comma_video_compression_challenge/pull/102)), and @rem2 ([#103](https://github.com/commaai/comma_video_compression_challenge/pull/103)). The HNeRV decoder, brotli outer compression, FP11 weight wrapper, and canonical Huffman for the latent sidecar are byte-identical / unchanged from @SajayR's PR #101 reference — these are the inherited substrate, not our contribution.

**Innovation classification** (per the operator-rigor question "which are NEW bolt-ons vs changes to existing PRs vs synergy from combining techniques"):

| # | Innovation | Classification | What it is + how it relates to PR #101 GOLD |
|---|---|---|---|
| 1 | **FEC6 31-mode frame-exploit selector** (K=16 active palette) | **NEW BOLT-ON** (no PR #101 equivalent) | A deterministic per-frame-pair transform space (identity / luma + RGB biases / blue-chroma amp / 1-pixel rolls). Offline (compress-time) scorer-targeted search picks one of K=16 transforms per pair against the upstream scorer's response on `videos/0.mkv`. Selector indices ship inside `x` (the rate-charged payload) and are replayed at inflate time without on-device search. PR #101 GOLD's `inflate.py` is 2,073 bytes and has no per-pair selector mechanism at all — the FES1→FEC6 selector framework and the 31-mode palette are entirely new code (`src/frame_selector.py`, 7,980 bytes). |
| 2 | **Fixed-Huffman k=16 codebook on selector indices** | **NEW BOLT-ON, sister technique to PR #101's canonical Huffman** | Static 16-symbol prefix code (lengths 2..8 bits) sized to the empirical mode-frequency distribution on the contest video. Compacts the 600-pair selector from a naive 300 B (4 bits/pair fixed) to 249 B wire format (~3.24 bits/pair empirical). PR #101 GOLD's `src/codec.py` already provides `decode_canonical_huffman` for the latent sidecar (adaptive, per-archive length-vector); we apply Huffman to a NEW layer (selector indices) with a FIXED code, so there are no per-archive header bytes spent declaring the code. The Huffman primitive is inherited; the layer and the fixed-table design are new. |

**Synergy:** the FEC6 selector indices live inside the same `x` blob that PR #101's existing brotli q=11 outer wraps — so the fixed-Huffman-compressed selector stream further benefits from outer LZ77 + entropy coding on top of the fixed-table Huffman. Net `archive_bytes_added = 178,517 − 178,258 = 259` over PR #101 GOLD's source archive: the rate term increases by `+25 × 259 / 37,545,489 ≈ +0.000172`, while the selector buys `−0.000794` on the CPU axis. Net `−0.000622` (the headline `−0.000794` minus the rate cost the selector itself pays).

**Inherited from PR #101 GOLD substrate (not our contribution):**

- **HNeRV decoder** — `src/model.py` is byte-identical to @SajayR's reference.
- **Brotli q=11 outer compression** of the state-dict + scale streams — PR #101 GOLD's `src/codec.py` already does `concatenated Brotli streams of q-bytes + fp16 scale per tensor`; we use this unchanged.
- **Canonical Huffman for the latent sidecar** (`u8 dim, i8 delta_x100` per pair) — PR #101 GOLD's `decode_canonical_huffman_all`; we use this unchanged via `src/codec_sidecar.py`.
- **FP11 weight wrapper** + magic-byte format — PR #101 GOLD's payload grammar; we use this unchanged.

Headline score against the current top merged submission ([@SajayR #101](https://github.com/commaai/comma_video_compression_challenge/pull/101), GOLD, `0.192845 [contest-CPU]`):

- `0.192051 [contest-CPU]` on Linux x86_64 via `upstream/evaluate.py --device cpu` — a `-0.000794` improvement on the axis the leaderboard ranks.
- `0.226210 [contest-CUDA T4]` on Modal T4 against the same archive bytes and runtime tree (canonical paired Modal A100 auth_eval; Modal call_id `fc-01KRMP4ZM5J8P1H3R2JN96VSC5`, 600 samples, 2026-05-15).

## Reproducibility

The runtime tree (`inflate.sh`, `inflate.py`, `src/codec.py`, `src/frame_selector.py`, `src/model.py`) lives alongside `archive.zip` in the submission directory per the upstream contract. ## Limitations

- Single-video, contest-runtime target. FEC6 is selected per-frame against the upstream scorer on the contest video; we do not claim generalization to unseen dashcam clips.
- We have not used the four-day-shutdown clause; the submission is competitive on the CPU axis against the current top, and we believe also innovative under the 2026-05-11 new-submission gate ([@YassineYousfi #108 closure](https://github.com/commaai/comma_video_compression_challenge/pull/108)).
- The `report.txt` preamble embeds an absolute path string. This is the verbatim format `upstream/evaluate.py` writes; it does not affect any scored value and is left as-emitted for parity with prior medal-class submissions.
