# is this submission competitive or innovative? explain why
<!-- competitive: better than top #1 submission -->
<!-- innovative: it has a novel idea that is not on the leaderboard yet, might not be competitive, but has potential -->

**Competitive: yes.** `0.192051 [contest-CPU]` vs PR [#101](https://github.com/commaai/comma_video_compression_challenge/pull/101) `hnerv_ft_microcodec` (`0.192840`, GOLD per @YassineYousfi's comment), delta `-0.000789`. The selector adds +259 bytes (rate `+25 × 259 / 37,545,489 = +0.000172`) and saves `-0.000961` across the distortion terms; distortion savings are **~5.6× the rate cost**.

**Innovative: yes.** Each frame has a different optimal perturbation against the upstream scorer. The scorer is also asymmetric across pair positions: SegNet reads only the second frame, PoseNet reads both. So frame-0 perturbations move PoseNet alone, frame-1 perturbations move both scorers, and no single perturbation wins across all 1200 frames. An offline sweep tool (`encoder/frame_exploit_segnet_posenet_sweep.py`, macOS CPU local) scores candidate per-frame perturbations (integer luma/chroma biases, single-pixel rolls, 8×8 signed chroma tiles, applied to either frame) against both scorers. The encoder (`encoder/build_pr101_frame_exploit_selector_packet.py`) picks K=16 modes from that table and ships the per-frame winner's index over a fixed Huffman codebook (~4 bits/frame). For this archive the K=16 winners are all frame-0; frame-1 candidates were swept but did not earn slots. Frame-0 is structurally invisible to SegNet, so frame-0 perturbations pay zero SegNet cost; that is the exploit this submission's name refers to. PR #101's weights reuse byte-identically; only the per-frame indices are transmitted. The sweep + selector pipeline applies to other frozen-substrate PRs.

FEC6 selector bolt-on around the public HNeRV lineage. Decoder (`src/model.py`) byte-identical to PR [#95](https://github.com/commaai/comma_video_compression_challenge/pull/95) (@AaronLeslie138). Byte substrate from PR [#101](https://github.com/commaai/comma_video_compression_challenge/pull/101) (@SajayR): compact decoder, latent-payload parsing, Brotli source streams, canonical Huffman on the latent sidecar (all reused unchanged). Member `x` grammar: `FP11 | u32 source_len | source_pr101_payload | u16 selector_len | selector_payload`. PR #101's payload reads verbatim from `source_pr101_payload`; the FEC6 selector is byte-appended outside the Brotli envelope and is not further compressed. Lineage: PR [#98](https://github.com/commaai/comma_video_compression_challenge/pull/98) (@EthanYangTW), PR [#100](https://github.com/commaai/comma_video_compression_challenge/pull/100) (@BradyMeighan), PR [#102](https://github.com/commaai/comma_video_compression_challenge/pull/102) (@EthanYangTW), PR [#103](https://github.com/commaai/comma_video_compression_challenge/pull/103) (@rem2). Paired Modal Tesla T4 against the same `archive.zip` bytes returned `0.226210 [contest-CUDA]`; CPU and CUDA scored separately on the same archive bytes. Decoded RGB frames are bit-stable per device but differ across CPU and CUDA because `F.interpolate(..., mode='bicubic')` and the `clamp/round/uint8` cast are not bit-identical across the two backends. Bundle ships `LICENSE` (MIT), `README.md`, `requirements.txt` (`torch`, `brotli`), `THIRD_PARTY_NOTICES.md`, `expected_output.sha256`, and a `tests/` regression for the canonical decode SHA. Research notes at [`adpena/comma-lab`](https://github.com/adpena/comma-lab); tooling at [`adpena/tac`](https://github.com/adpena/tac).
