## Pattern: ae430d78 audit's failure mode

The ae430d78 subagent (PR107 hardening) generated all 8 plans by applying generic templates ({existing codec X} → {PR106 component Y}) without verifying:
1. Whether PR106 has component Y at all (#05, #06: PR106 has no mask channel)
2. Whether codec X accepts PR106's data distribution / state-dict schema (#02 #03 #04)
3. Whether codec X's information-theoretic ceiling exceeds the existing brotli baseline (#02)

Lane Ω-W-V3 (#01) is the standout because `tac.water_filling_codec_v2` was designed schema-agnostic from the start: it operates per-Conv2d-tensor with per-channel block-FP-with-Hessian-aware-bit-allocation. The PR106-applicability question reduces to "is the per-tensor contract met" (yes for all 13 Conv2d weights in PR106) rather than "does codec X know about PR106's specific architecture".

## Pattern: high-leverage prevention rule

For any future "X codec → Y component" stacking proposal generated by a subagent:
- BEFORE recommending dispatch, run a local smoke test:
  1. Does codec X's API accept Y's actual state dict / data tensor without ValueError?
  2. Does codec X's data-distribution assumption (e.g., "ternary regularized weights", "discrete class output") match Y's actual data?
  3. Is codec X's information-theoretic ceiling (0-th-order Shannon entropy) STRICTLY BELOW the existing baseline encoding's effective rate?
- If ANY of these fail, mark the lane FALSIFIED before any GPU dispatch.

This rule could be wired as a preflight check on any new revival_plan_*.md memory file.
