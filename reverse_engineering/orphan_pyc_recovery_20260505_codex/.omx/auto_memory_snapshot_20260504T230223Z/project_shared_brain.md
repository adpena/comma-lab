## Architecture

The DuckDB at `.omx/state/review_tracker.duckdb` is the canonical data store for ALL project state:

### Tables
- `entities` — 2,120 AST-parsed code entities with review status
- `reviews` — audit log of all review actions
- `findings` — research findings (discoveries, negatives, techniques)
- `runs` — training run results (proxy/auth scores, distortions)
- `council_outputs` — council session outputs (strategic, review, signal_loss)
- `scan_meta` — scanner metadata

### Auto-writers (in `src/tac/utils.py`)
- `record_finding(category, title, body, score, variant, tags)` — dual-writes to DuckDB + findings.md
- `record_run(variant, platform, epoch, scores...)` — dual-writes to DuckDB + run_log.md
- `record_council(session_type, title, body, file_path)` — DuckDB only

### Query interface
- `python tools/review_tracker.py query "SELECT ..."` — read-only SQL
- `python tools/review_tracker.py dashboard` — color-coded terminal dashboard
- `python tools/review_tracker.py policy-check <file>` — policy compliance

### Pre-commit hook
- Automatically runs on every `git commit`
- Shows color-coded review status of staged files
- Currently warn-only; set `REVIEW_GATE_WARN_ONLY=0` to enforce

**Why:** Prevents signal loss. All findings, runs, and council outputs are queryable from any conversation. Markdown files are the human-readable mirror, auto-generated by dual-write.

**How to apply:** Wire `record_finding()` and `record_run()` into training scripts' best-checkpoint and eval-complete events. Use `record_council()` when saving council outputs instead of just writing to ~/Downloads/.
