```bash
  --public-scan-path submissions/pr106_latent_sidecar_r2/archive.zip \
  --public-scan-path submissions/pr106_latent_sidecar_r2/inflate.sh \
  --public-scan-path submissions/pr106_latent_sidecar_r2/inflate.py \
  --public-scan-path submissions/pr106_latent_sidecar_r2/src \
  --public-scan-path submissions/pr106_latent_sidecar_r2/README.md \
  --public-scan-path submissions/pr106_latent_sidecar_r2/report.txt \
  --public-scan-path submissions/pr106_latent_sidecar_r2/archive_manifest.json \
  --public-scan-path submissions/pr106_latent_sidecar_r2/contest_auth_eval.json \
  --json-out submissions/pr106_latent_sidecar_r2/pre_submission_compliance.contest_final.json \
  --strict
```

**Why scoped public-scan-paths?** The internal custody/compliance JSONs
(`pre_submission_compliance.contest_final.json`, `pre_submission_compliance.custody_packet.json`)
serialize operator-local absolute paths (`runtime_root`, dispatch-claim ledger row notes) and the
auth-eval `contest_auth_eval.json` may also embed Modal-side absolute paths. Scoping
`--public-scan-path` to the OSS-publishable subset keeps the check idempotent across re-runs
without sacrificing the public-hygiene contract on the actual release surface (archive +
runtime tree + report + manifest + README + auth-eval custody).
